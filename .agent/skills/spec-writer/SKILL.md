# Spec Writer (sansu) - Workspace Skill

このスキルは「仕様書に反映する内容を網羅的にまとめて」「仕様書用に整理して」「app_specに追記」などの要求が出たときに適用する。

## 1) ゴール
- ユーザーの会話ログから、仕様書へ追記すべき内容を **漏れなく** 抽出する
- 仕様書に貼れる **完成Markdown** で出す
- 用語の揺れを検出し、統一案を提示する（勝手に変更はしない）

## 2) 出力フォーマット（固定）
以下のセクションを順に出す（該当がない場合は「なし」で省略しない）：

## 追加/更新ポイント一覧
- (箇条書きで短く)

## 仕様書追記（コピペ用）
### 1. 画面/ナビ
### 2. 学習フロー
### 3. 出題/採点/復習ロジック
### 4. データ構造（保存/同期）
### 5. テスト（アプリ内/印刷PDF）
### 6. 文言・トーン（正解/不正解/ヒント）
### 7. 設定項目
### 8. 運用（保護者導線・エラー時）
### 9. 未決事項/要確認（質問ではなく列挙）

※ 各項目は「仕様」と「理由」をセットで書く：
- 仕様：〜する
- 理由：〜だから

## 用語集（追記/更新）
| 用語 | 意味 | 表示名（UI） | 内部名（コード） | 備考 |
|---|---|---|---|---|

## チェックリスト（抜け漏れ防止）
- [ ] 既存仕様と矛盾していない
- [ ] 例外ケースを考慮した
- [ ] UI文言トーンが統一されている
- [ ] 画面遷移が明確
- [ ] 進捗/記録/設定の導線が明確

## 3) 抽出ルール
- ユーザーが「決めた」「Bで」「これでいく」「おすすめはこれ」など合意した内容は **確定仕様** として書く
- 「検討」「あとで」「とりあえず」などは **未決事項** に入れる
- 数値・回数制限（例：1問につきスキップ1回）などは最優先で抽出する

## 4) 用語統一ルール
- 既存用語（due/weak等）を尊重し、変更は提案に留める
- UI表示名と内部名を分けて表で管理する

## 5) 禁止
- 仕様を勝手に増やさない
- 断定できないことを確定仕様にしない
- ユーザー未合意の命名変更を既存仕様に混ぜない
