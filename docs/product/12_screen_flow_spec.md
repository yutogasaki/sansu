# docs/product/12_screen_flow_spec.md — 画面・遷移仕様

> 状態: gameplayは **MVP-0/1**、run・回答receipt・終了status保存は **MVP-2a**、Study共通planner / SRS接続は **MVP-2b**。`/` は `/explore` へ転送し、説明CTAを挟まない探索ランを通常起動面とする。`/battle` は探索優先の「探検基地」へ再編し、既存の学習・2人ゲーム・設定ルートは維持する。

## 1. ルート方針

### 1.1 現行ルート

| ルート | 分類 | 方針 |
|---|---|---|
| `/` | 起動エイリアス | `/explore` へ `replace` 転送する。初回設定完了後もこの経路を使う |
| `/battle` | 探検基地 | 探索からの退出先。相棒と次の未発見を予告し、1タップで再出発できる。2人ゲームも下位導線から引き続き開ける |
| `/battle/play` | 現行維持 | 2人ゲーム本体 |
| `/explore` | 通常起動面 | 探索ラン本体。説明CTAなしで全画面表示し、下部ナビを隠す |
| `/study` | 維持 | 復習、定期テスト、英語、開発確認を行う既存学習 |
| `/stats` | 現行維持 | 既存の学習記録。発見ノートはまだ統合しない |
| `/settings` | 維持 | プロフィール、音、学習設定 |
| `/parents` | 現行維持 | 保護者向け画面 |
| `/dev` | 現行維持 | 問題確認・開発用画面 |

現行の起動フロー:

```text
通常起動 / 初回設定完了 → / → /explore（探索ランを直接開始）
探索から退出                       → /battle（探検基地）
```

`/study`、`/settings`、`/stats`、`/battle` は独立ルートとして残し、起動面の置換を理由に削除・移行しない。

### 1.2 後続の統合目標

| 段階 | ルート方針 |
|---|---|
| 起動面統合済み | `/` から `/explore` へ直接転送する |
| MVP-3 | 探索開始面へ、いきものの探索リアクションを統合する |
| MVP-3 | `/stats` に発見ノートを追加し、学習記録と同居させる |
| 統合済み | `/battle` を退出後の探検基地にし、`/explore` の通常起動とは役割を分ける |
| MVP-4 | `/study` は復習・テスト・英語用として維持する |
| 統合済み | 下部 `Game` を `基地` へ改名し、既存2人ゲームは基地内の下位導線として維持する |

## 2. 探検基地（現行）

目的: 子どもが1タップで探検を始めたくなる。

> 通常起動は `/explore` へ直接入り、探検基地は探索から帰った後や下部ナビから開く再出発面とする。説明を読ませて通常起動を遅くしない。

表示要素:

- 相棒と、次の探索に一つだけ残った未発見の気配。cold-open価値ゲート未通過のactor、固有名、図鑑進捗を恒久要素として置かない
- 画面上部で完全に読める68px以上の主CTA「すぐ たんけん」
- 主CTAから `/explore` へ1タップで再出発
- 既存の協力・つなひき・詳細設定は「ふたりで あそぶ」の下位カードへまとめる
- 架空の進捗、未達成バッジ、倒した敵、絵文字だけの装飾は置かない

文言例:

```text
ちていで なにかが まってる
つぎは どんな うごきかな？
[すぐ たんけん]
```

## 3. 探索開始

- 通常起動では独立した説明画面、開始CTA、ゲーム選択を置かない
- プロフィールとrun保存の解決後、固定cold-openは道選択を挟まず0タップで最初の問題へ入る。選択内容を反映できない行き先カードを先に見せない
- 最初の意味ある道選択はcold-openの3問完了後に置き、選んだnodeの地形・操作・問題画面を一致させる

## 4. 探索ラン画面

スマホ縦を第一に、タブレット横にも対応。

縦画面構成:

```text
┌──────────────┐
│ 帰還 / エネルギー / 発見数        │
├──────────────┤
│ 道選択中: 現在地と次の道だけ      │
│ 問題中: 現在地→対象だけの簡易地図 │
├──────────────┤
│ 道カード / 橋選択                │
│ または 問題 + テンキー           │
└──────────────┘
```

重要:

- 地図と問題を同時に複雑表示しない
- スマホの道選択では通過済み・未到達ノードを隠し、現在地と今回選べる次の道だけを見せる。全体マップはタブレット以上または専用の地図面へ限定する
- 子どもの主操作は常に明確に1つ
- 通常発見と通常調査手掛かりでは問題UIを隠さない。大発見だけは問題区間を終えて全画面へ切り替えてよい
- 一度道を選んだ後は3問を1区間とし、区間内の次ノードと橋プランは決定論的に自動選択する
- active runのreloadは説明画面や「続ける」確認を挟まず、同じroute、energy、finds、式、attemptNumberへ直接戻す。入力途中の数字だけを空にし、保存済み回答をもう一度解かせない
- 未確認のQ7大発見はreload後に1回だけ再表示する。確認操作のcheckpoint保存が成功した後は、reloadしても大発見modalを再掲せずQ8へ戻す。通常toastの再表示可否をcomponent refだけで決めない
- checkpointの復元・tail replay・保存に失敗した場合は問題を推測して進めず、現在のProblemを保った責めない再試行面を出す。旧active runにcheckpointがない場合だけ、安全にabandonedとして新しいrunへ切り替える
- delivery / feature-flag ID `snap-root-v1` のlocal validationへ載せる現行visual candidate `dig-pop-painted-v2` は、同じカメラ、相棒、スコップ、土、根生物を保つ。commit済み正解数に応じて `ready → dig-one → dig-two → popped` と進み、1・2問目は続行操作を置かずsubmitから650ms以内に次問を操作可能にする。相棒が根生物の身体や葉を引かず同じ土を掘り、土が飛ぶ、根生物の足が見える、全身でぽんと抜ける進行を見せ、3問目は相棒の安全な尻もちと柔らかい土塊の葉帽子をinlineで見せる。v2はlocal runtimeへ配線済みで、mobile / tablet視覚ゲートを通過したが、残る非補償ゲートが通るまでproductionへ昇格しない
- cold-openの問題式と入力は通常のDOMを使い、演出のために問題の値・演算・category・assignment・SRS扱いを変更しない。誤答ではstageと同じ問題を維持し、submitから550ms以内を目標に再入力可能にする。支援先読みがtimeoutまたはerrorになった場合も、遅れて問題を差し替えず同じ入力面へ戻す
- 1・2問目のopening途中発見はtoastを抑止し、問題、TenKey、身体変化を同時に読める状態を維持する。3問目も同じ画像を結果modalへ再掲せず、説明CTAなしで次の道選択へ進める
- production defaultは `classic-v1` とする。旧編み根版Snap Rootとそのpayoff variantは50 / 100のREJECTであり、localを含む安全候補から外す。一本葉を引くBloom版と水やり版は同じ `snap-root-v1` slotの旧visual candidateとしてHOLDかつ非採用とする。`dig-pop-painted-v1` はsource HOLDの履歴、`dig-pop-painted-v2` はlocal validation candidate、`root-pull-v1` / `root-pull-v2` は比較用presentationとし、production価値ゲート通過前は基地、きろく、永続図鑑catalogへ固定しない

## 5. 問題表示

問題そのものは見やすく、演出は周辺で行う。

NG:

```text
この水晶に秘められし数の鍵を解き放て！
```

OK:

```text
7 + 8 = ?
```

周辺演出:

```text
水晶をひらこう
```

### 5.1 保存待ちと再試行

- `/explore` で出発した後、プロフィール解決とrun開始保存が完了するまで道・問題操作を解禁しない。待機中は「たんけんノートを じゅんびしているよ」と短く伝える
- 回答送信後はテンキー、Enter、帰還を一時的に無効にする。ただし保存表示は250msを超えた場合だけ問題と同じ領域へ出し、高速な正常保存で事務処理を見せない
- 保存成功receiptを受け取る前に正解・不正解演出、エネルギー、attemptCount、発見、問題更新を進めない
- 回答保存に失敗した場合は、同じ式、入力済みanswer、AttemptIdentity、attemptCount、energyを維持する。問題領域内で「ノートに まだ かけなかったよ。こたえは そのままです。」と伝え、44px以上の「もういちど きろくする」を置く
- 高速なclick / Enterの重複を含め、保存中の同じ回答は1件だけとして扱う。retryは新しいattemptを作らず、最初に凍結したidentityを再送する
- 帰還・救出はrun終了保存後だけ帰還画面へ移る。終了保存失敗時は「いまの たんけんは そのままです」と伝え、status、確定発見、画面をterminalへ進めず同じ終了操作を再試行する
- 最後のひかりで発見した場合は、発見dialogを読み「調査をつづける」で閉じるまでrunを `active` のまま保つ。その後に `rescued` を保存し、成功後だけ帰還画面へ進む
- 文言は通信・正誤・子どもの操作を責めず、現在維持されているものと次の安全な操作を1文ずつで伝える

## 6. 発見画面

発見は「世界で起きた変化」と「調査ページに増えた性質」を同じ因果として見せる。

```text
あおい水晶を見つけた！
```

- 通常発見は約900msの自動トーストとし、表示中も次問または次の区間選択を操作できる
- 調査ページに属さないレア標本も約900msの非blocking toastとし、希少度だけでは続行操作を要求しない。調査ページの通常手掛かりも標本希少度に関係なく非blockingとする
- Golden Discovery Page の特徴発見は、開いた調査ノートに3つの手掛かりスロットを示し、今回埋まった1件と次の未発見ヒント1件だけを読めるようにする
- Golden Discovery Page の通常特徴も約900msの非blocking toastとし、今回の発見文を先に見せ、詳細な3スロットと次の手掛かりは帰還時の調査ページで再読できるようにする
- 定義済みの特徴連鎖が完成した場合は、希少度にかかわらず「大発見」へ切り替え、明示的な続行操作を置く
- `dig-pop-painted-v2` の1・2問目はopening途中発見のtoastを出さず、問題UIを隠さない。3問目の `popped` は同じ画像を別の結果モーダルへ再掲せず、説明用CTAを挟まず次の道選択へ進める。大発見・固有名・図鑑遷移はcold-openへ追加しない
- root-tangleの大発見は、保存済み回答による開通・相棒通過を見せた同じcameraを紙・濃線の観察画へ変換し、その上へ因果文と3つの前提手掛かりを置く。別の汎用挿絵へ切り替えない
- 観察画はcommit済みattempt key、node、encounter、直近discovery、最終featureが一致する場合だけ開く。保存中、保存失敗、別node、古いreaction、前提不足では開かない
- 大発見の表示中だけ背後の探索UIを `inert` にする。通常標本、レア標本、通常調査手掛かりのtoastは、標本希少度に関係なく背後を操作可能に保つ
- 発見名、開いた特徴数、次の手掛かりはDOMテキストとARIAで伝え、画像だけへ焼き込まない
- 大発見のダイアログは続行ボタンへフォーカスし、フォーカスを閉じ込める。Esc、または続行ボタン上のnative Enterでも続行できる。通常標本、レア標本、通常調査手掛かりはフォーカスを移動しない
- root観察の続行操作は56px以上、補助操作は44px以上、意味を持つ文言は12px以上とし、390×844の初期viewportから到達できるようにする
- `prefers-reduced-motion` では移動・拡縮を止め、短いフェードまたは即時更新にする

## 7. 帰還画面

1プレイの物語を見せる。Golden Discovery Page があるランは、帰還直後を一時的な Research Library として構成し、「今回の1冊が育った」ことと「次の未知へ出発する」ことを同じ空間でつなぐ。

```text
今日の探検
- あおい水晶 ×2
- 葉っぱの化石 ×1
- ひみつの道を発見

あと1マスで虹の部屋だった…！
[もう一回] [ホーム]
```

「正解数」だけで終わらせない。

### 7.1 Research Library 帰還縦切り

- DOMの主順序は「今回の調査本 → 次の探検を始める主操作 → 標本・次の気配・退出などの補助情報」とする
- 調査本には、今回のランで最後に進めた `researchPage` 1件だけを表示する。検証中のcold-openを仮の生態メモとして選ばず、ほたる花を進めた場合は最新の主調査を選ぶ。将来、価値ゲートを通ったcold-openページを追加する場合も、古い完了ページではなく最後に進めた主調査を優先する。ページ名、最終特徴の `finding`、手掛かり状態はDOMとARIAで伝え、挿絵へ焼き込まない
- 主操作はamber系の物理レバーとして見せ、56px以上の明示的なボタンにする。390×844では主操作の下端を620px以内へ置くことを目標にする
- `replayTeaser` は未訪問の場所へ続く「出会えるかも」として1件だけ見せる。帰還画面の再出発CTAや再出発直後のcold-openが、その場所への到達を保証するような断定文にはしない。実際の道選択は次の3問区間境界で行う
- 帰還画面はライブ遭遇より静かにしてよいが、旧目安「不可思議な図鑑 70 : ポップ 30」は使わない。深い青緑は本を囲む奥行きとoutlineへ抑え、完成した発見、次へ動く相棒、主レバーだけをcoral / amber / turquoiseの大きな色面で強調する
- 390pxの3秒テストでは「次へ行きたがる相棒 → 今回変化したページ対象 → 次に押すamberレバー」の順に指せること。意味を持つscene copyは12px以上にする
- reduced motionでは本・経路・相棒の移動や拡縮を止め、静止状態または短いフェードで同じ因果と次操作を伝える
- 200%文字拡大や短い画面では縦スクロールを許可するが、主操作は補助情報より前のDOM順を維持する

MVP-0/1では次を表示しない。

- catalogに存在しない複数ページ棚、名前付き未発見ページ、収集率、永続的なNEW表示、前回との増減
- 同じ `onRestart` に接続する押下可能な2分岐
- ランをまたいで調査ページやスタンプが保存されると誤解させる文言

無名のページ端、奥の扉、分岐線は、非操作・`aria-hidden` の世界装飾としてのみ使ってよい。button枠、押下影、focus state、pointer cursorを与えない。複数ページのLibrary一覧はMVP-2の発見feature永続化後に別途実装する。

## 8. きろく / 発見ノート（MVP-3目標）

統合表示:

- 学習: 今日の問題数、復習状況、できるようになったスキル
- 探索: 発見図鑑、最近の探検ログ、未発見エリア

子どもには発見中心。保護者には学習中心。

## 9. 保護者画面

保護者には以下を明快に説明する。

- 探索は算数問題を解くことで進む
- 現行MVP-2bの問題はStudy共通plannerで選び、プロフィールに近い既存の単一数字入力問題ジェネレーターから作る
- runへ予約したassignmentと実出題skillが一致する回答だけ、SRS・回答ログ・弱点・解放判定へ原子的に反映する
- 未対応入力や生成不能のgame-only fallbackは探索eventだけへ残し、学習進捗へ混ぜない
- ゲーム上の失敗はラン内のエネルギー・コンボに限定
- ランキングや課金圧力はない
